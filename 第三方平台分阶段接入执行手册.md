## 2. 接入顺序（按优先级）
1. 阶段A：设备接入平台（EMQX Cloud 或 ThingsBoard）
2. 阶段B：天气与灾害数据平台（和风天气 QWeather 或同类气象接口）
3. 阶段C：地图与轨迹平台（高德开放平台 AMap）
4. 阶段D：区块链存证平台（蚂蚁链开放服务 或 腾讯至信链）
5. 阶段E：多模型AI平台（火山方舟/百度千帆/阿里百炼，作为DeepSeek并行）
6. 阶段F：企业消息通知平台（企业微信/钉钉）

## 3. 通用执行原则
- 先打通最小可用链路，再扩展能力字段。
- 每阶段完成后必须进行联调验收，不跨阶段堆积问题。
- 涉及密钥或回调地址变更时，优先使用环境变量与配置中心，不写入代码仓库。
- 所有外部调用必须具备超时、重试、熔断和失败降级策略。

## 4. 分阶段执行清单

### 阶段A：设备接入平台（P0）
目标：统一多厂商设备协议，稳定写入现有传感器接入链路。

实施步骤：
1. 确定主接入平台（EMQX Cloud 或 ThingsBoard 二选一）。
2. 建立测试租户、产品、设备模板与Topic命名规范。
3. 配置平台到本系统的上行转发（HTTP webhook 或 MQTT桥接）。
4. 完成设备鉴权映射（平台设备ID -> 本系统deviceCode）。
5. 完成联调：温湿度、土壤、光照三个样本设备持续上报。
6. 验证告警联动与设备在线状态回写。

暂停点A1（需人工）：创建平台账号并完成实名/企业认证。
你需要做什么：
- 注册并登录目标平台控制台。
- 完成企业认证（如平台要求）。
- 将控制台管理员账号准备给实施人员用于后续配置。
- 完成后请回复：继续阶段A（A1已完成）

暂停点A2（需人工）：申请生产级连接配额与公网白名单。
你需要做什么：
- 在平台侧申请实例规格升级或并发上限。
- 提交本系统公网出口IP做白名单。
- 确认平台工单审批通过后再继续。
- 完成后请回复：继续阶段A（A2已完成）

A2提交清单（建议一次性准备，避免来回补资料）：
- EMQX实例ID/名称
- 目标环境（测试/预发/生产）
- 预计设备并发数、消息TPS、峰值上报频率
- 本系统公网出口IP（如有多出口需全部列出）
- Webhook回调地址：`/agri/envSensor/ingest/emqx` 完整公网URL
- 鉴权Header名称与Token方案（建议沿用现有X-Agri-Token）

阶段验收标准：
- 连续24小时设备上报成功率 >= 99%。
- 平台断连后可自动恢复，且本系统无大面积丢数。

### 阶段B：天气与灾害数据平台（P0）
目标：为病虫害预警、产量预测与物流预警提供外部气象特征。

实施步骤：
1. 申请天气API Key并确定调用城市/地块坐标映射规则。
2. 接入实时天气、逐小时预报、降雨预警三个接口。
3. 建立天气数据缓存层（建议5-15分钟TTL）。
4. 将天气特征注入现有预测与预警接口。
5. 完成有天气特征/无天气特征的A/B输出对比。

当前进展：
- 已完成第三方API统一接入页的天气探活入口。
- 已完成后端天气配置与天气摘要解析，支持先用探活确认联通性。
- 已完成后端编译与前端构建验证。

暂停点B1（需人工）：开通天气平台商业套餐。
你需要做什么：
- 在控制台选择满足调用量的套餐。
- 绑定支付方式并开通正式环境Key。
- 将正式Key放入服务器环境变量，不通过聊天工具传递。

阶段验收标准：
- 天气接口调用失败时，系统仍可用且自动降级为“仅内部数据预测”。
- 关键预测页面响应不因天气接口显著变慢。

### 阶段C：地图与轨迹平台（P1）
目标：增强物流轨迹回放、异常停留识别与可视化能力。

实施步骤：
1. 开通Web端和服务端Key，配置安全域名/签名。
2. 接入轨迹绘制、逆地理编码、路径纠偏能力。
3. 在物流模块增加电子围栏与异常停留判定。
4. 完成轨迹回放与告警点标注展示。

当前进展：
- 已完成物流路径追踪页的轨迹地图预览骨架，可直接从现有物流记录查看路线节点与坐标信息。
- 已完成后端地图平台配置骨架，待人工补齐 AMap Key 与安全配置后接入实网地图能力。
- 已完成后端编译与前端构建验证。

暂停点C1（需人工）：域名备案与安全白名单配置。
你需要做什么：
- 提供前端正式域名并完成备案要求（如需要）。
- 在地图平台控制台配置允许来源域名与IP。
- 提供安全策略截图用于验收归档。

阶段验收标准：
- 轨迹回放与地图点位对应准确，无明显漂移。
- 异常停留告警与时间线展示一致。

### 阶段D：区块链存证平台（P1）
目标：实现低成本、可核验的数据存证能力。

实施步骤：
1. 确定“链下明细 + 链上哈希”存证方案。
2. 接入存证上链API与回执查询API。
3. 建立批量存证任务与失败重试队列。
4. 在溯源详情页展示存证哈希与校验结果。

暂停点D1（需人工）：法务与合规确认。
你需要做什么：
- 由法务确认存证数据字段的合规边界。
- 明确哪些字段可上链、哪些必须脱敏或只保留哈希。
- 形成签字版合规清单后再继续开发。

阶段验收标准：
- 任意抽样记录可完成“本地数据 -> 哈希 -> 链上回执”核验闭环。
- 单条存证成本与时延在预算范围内。

### 阶段E：多模型AI平台并行接入（P1）
目标：构建多模型容灾与成本优化能力，避免单平台依赖。

实施步骤：
1. 统一AI网关抽象（模型路由、超时、重试、回退）。
2. 接入第二模型平台并完成同一提示词对照测试。
3. 配置按场景路由（识别、预测、摘要）与限流策略。
4. 建立模型效果与成本看板。

暂停点E1（需人工）：开通企业级模型配额与合同。
你需要做什么：
- 完成供应商商务签约与发票信息确认。
- 确认模型调用地域、数据留存策略与SLA条款。
- 提供合同编号给项目组做合规归档。

阶段验收标准：
- 主模型失败时可在秒级切换到备模型。
- 模型调用成本可按业务线统计并可追溯。

### 阶段F：企业消息通知平台（P2）
目标：将预警与待办闭环推进到责任人。

实施步骤：
1. 接入机器人Webhook或应用消息通道。
2. 建立告警模板（设备异常、预测偏差、在途异常）。
3. 配置告警分级与通知策略（立即/汇总/静默时段）。
4. 打通“消息 -> 待办任务 -> 处理结果回写”链路。

暂停点F1（需人工）：组织通讯录与角色映射。
你需要做什么：
- 导出并确认部门、岗位、负责人映射关系。
- 确认哪些角色接收P0/P1/P2级告警。
- 完成消息接收范围审批。

阶段验收标准：
- P0告警5分钟内触达责任人。
- 消息点击后可定位到系统对应处理页面。
