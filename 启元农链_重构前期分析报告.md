# 启元农链重构前期分析报告

本报告基于当前仓库只读扫描生成，覆盖后端 `src/main/java`、前端 `src/`、重复代码、空壳功能与架构风险评估。

## 1. 范围与结论

- 后端以 Spring Boot + MyBatis + 若依框架为主，Agri 业务模块已形成完整 CRUD 与若干运营看板接口，但大量控制器与服务仍保留模板化结构。
- 前端以 Vue2 + Vuex 为主，页面数量多，业务页普遍采用同构 CRUD 模板，重复方法和重复样式明显。
- 架构上最突出的风险是 Controller 过厚、Service 贫血、前端页面过大且状态分散。

## 4. 重复与相似代码

### 4.1 Controller 重复

- 30 个 Agri Controller 几乎都使用同一 CRUD 模板：`list / export / getInfo / add / edit / remove`。
- 典型证据： [AgriBaseRuleController.java](RuoYi-Vue-master/ruoyi-admin/src/main/java/com/ruoyi/web/controller/agri/AgriBaseRuleController.java#L42)、[AgriMarketForecastController.java](RuoYi-Vue-master/ruoyi-admin/src/main/java/com/ruoyi/web/controller/agri/AgriMarketForecastController.java#L52)、[AgriTodoTaskReminderController.java](RuoYi-Vue-master/ruoyi-admin/src/main/java/com/ruoyi/web/controller/agri/AgriTodoTaskReminderController.java#L51)
- 看板端点也高度同构：`/dashboard`、`/dashboard/ops` 在多个控制器中重复出现。

### 4.2 Service 重复

- 多数 Agri ServiceImpl 仅做 Mapper 委托，缺少聚合、事务边界与领域规则。
- 典型证据： [AgriMarketForecastServiceImpl.java](RuoYi-Vue-master/ruoyi-system/src/main/java/com/ruoyi/system/service/impl/AgriMarketForecastServiceImpl.java#L16)、[AgriDataUplinkTaskServiceImpl.java](RuoYi-Vue-master/ruoyi-system/src/main/java/com/ruoyi/system/service/impl/AgriDataUplinkTaskServiceImpl.java#L16)
- 默认值归一化逻辑在多个服务中重复出现，例如 [AgriCarbonFootprintModelServiceImpl.java](RuoYi-Vue-master/ruoyi-system/src/main/java/com/ruoyi/system/service/impl/AgriCarbonFootprintModelServiceImpl.java#L655) 与 [AgriDeviceStatusMonitorServiceImpl.java](RuoYi-Vue-master/ruoyi-system/src/main/java/com/ruoyi/system/service/impl/AgriDeviceStatusMonitorServiceImpl.java#L103)

### 4.3 前端重复

- 大量页面都采用相同的 app-container + 查询区 + 表格 + 分页 + 弹窗结构。
- 典型证据： [views/system/user/index.vue](RuoYi-Vue-master/ruoyi-ui/src/views/system/user/index.vue#L2)、[views/agri/baseRule/index.vue](RuoYi-Vue-master/ruoyi-ui/src/views/agri/baseRule/index.vue#L2)、[views/agri/logisticsTrack/index.vue](RuoYi-Vue-master/ruoyi-ui/src/views/agri/logisticsTrack/index.vue#L2)

## 5. 空壳 / 弱实现

### 5.1 后端

- 业务主链路未见大量显式 TODO 占位，但存在若干 `return null` 的防御式兜底，需要区分“空壳”与“兜底”。
- 证据： [TokenService.java](RuoYi-Vue-master/ruoyi-framework/src/main/java/com/ruoyi/framework/web/service/TokenService.java#L82)、[AgriHttpIntegrationClient.java](RuoYi-Vue-master/ruoyi-system/src/main/java/com/ruoyi/system/integration/AgriHttpIntegrationClient.java#L825)
- `AgriEnvSensorRecordController` 与 `AgriYieldForecastTaskController` 内部存在大量解析/转换辅助函数，说明业务逻辑较多，但不属于空壳实现。

### 5.2 前端

- 两个临时验收页更接近展示壳，联调痕迹较弱： [tempLogisticsTrace/index.vue](RuoYi-Vue-master/ruoyi-ui/src/views/agri/tempLogisticsTrace/index.vue#L1)、[tempConsumerTrace/index.vue](RuoYi-Vue-master/ruoyi-ui/src/views/agri/tempConsumerTrace/index.vue#L1)
- 其余 Agri 正式页面普遍已绑定 API 与交互方法，未发现大面积空方法。

## 6. 架构问题评估

1. Controller 过厚：部分控制器行数超过 600，最高达到 865 行，业务计算、格式转换、异常兜底混入控制层。
   - 证据： [AgriYieldForecastTaskController.java](RuoYi-Vue-master/ruoyi-admin/src/main/java/com/ruoyi/web/controller/agri/AgriYieldForecastTaskController.java#L50)
   - 证据： [AgriEnvSensorRecordController.java](RuoYi-Vue-master/ruoyi-admin/src/main/java/com/ruoyi/web/controller/agri/AgriEnvSensorRecordController.java#L49)
2. Service 贫血：大量 Agri ServiceImpl 只是 Mapper 转发，缺少领域封装。
3. 异常治理分散：存在全局异常处理器，但业务层仍有本地 try/catch 风格不统一的问题。
   - 证据： [GlobalExceptionHandler.java](RuoYi-Vue-master/ruoyi-framework/src/main/java/com/ruoyi/framework/web/exception/GlobalExceptionHandler.java#L28)
4. 数据权限不一致：系统服务有 `@DataScope`，Agri 服务未见同级别约束。
   - 证据： [SysUserServiceImpl.java](RuoYi-Vue-master/ruoyi-system/src/main/java/com/ruoyi/system/service/impl/SysUserServiceImpl.java#L76)
5. 前端状态管理集中在单文件组件，未形成统一域模型或共享 store 结构。
6. 前端页面体量过大：多个 Agri 页面超过 600 行，最高接近 1000 行，维护成本高。

## 7. 优先重构建议

1. 先抽象 Agri CRUD 公共能力，压缩 Controller 重复。
2. 将看板/计算/格式化逻辑下沉到 Service 或 Domain Service。
3. 给前端 CRUD 页面提取通用组合逻辑，减少 `handleQuery` 等重复方法。
4. 统一 API 参数构造与错误处理。
5. 给 Agri 模块补齐数据权限、异常规范与领域边界。

## 8. 可小步执行的重构 + 增强计划

## 9. 可自然延伸的新功能候选

以下功能不作为当前强制范围，建议在前述重构稳定后择优启用：

1. 运营总览数据看板增强：把各模块的智能摘要、风险队列、待办闭环统一到一个首页工作台。
   - 后端需要：新增聚合查询接口，汇总多个 controller 的看板数据与 AI 摘要。
   - 前端需要：新增或重构首页卡片区、趋势图、风险列表和快捷入口。
   - 价值：把分散在各业务页的洞察收口为统一经营视图，便于管理层巡检。

2. 批量操作与批量审核：为高频台账页补齐批量导入、批量发布、批量审核、批量删除等能力。
   - 后端需要：补充批量接口、统一批处理结果返回、保留部分失败明细。
   - 前端需要：在列表页增加多选、批处理按钮、进度与失败反馈。
   - 价值：降低重复录入和逐条审核成本，适合台账密集型模块。

3. 审计日志与操作追踪看板：把关键智能操作、人工修改、异常回退统一沉淀为审计事件。
   - 后端需要：在智能调用、重要写回和导出路径补审计埋点，新增审计查询接口。
   - 前端需要：新增审计日志页、事件筛选、详情弹窗与导出入口。
   - 价值：提高可追责性，也能为后续问题复盘和权限治理提供证据。

如果你确认启用其中某一项，我会按同样的“小步、低风险、最多 3 个文件一组”的方式继续拆解成可执行任务单。
