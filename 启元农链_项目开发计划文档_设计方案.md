# 启元农链项目开发计划文档设计方案

## 1. 文档目标
本设计方案用于定义“项目开发计划文档”的结构、更新机制和跟踪口径，确保项目可以按阶段推进、按模块验收、按风险闭环。

适用项目：启元农链——AI+大数据驱动的区块链农产品溯源系统。

## 1.1 文档边界
- 本文件只描述计划文档应该怎么组织、怎么更新、怎么验收，不承载实时执行进度。
- 所有阶段状态、完成度、实际结束时间、验收结论都应落在正式计划文档中。
- 当结构、字段或口径发生变化时，先更新本设计方案，再同步调整正式计划文档。

## 2. 使用原则
- 以“可实现、可验收、可追踪”为第一原则。
- 以阶段推进为主线，以功能模块为最小跟踪单元。
- 每个功能必须具备：负责人、计划时间、状态、验收标准。
- 状态更新应在任务完成当天同步到计划文档。
- 风险、变更、延期必须记录原因与措施。

## 3. 文档结构设计

### 3.1 封面信息
建议字段：
- 项目名称
- 文档名称（项目开发计划）
- 版本号
- 当前状态（草案/执行中/封版）
- 编制人
- 最后更新时间

### 3.2 项目目标与范围
建议包含：
- 项目总目标
- 本期范围（MVP）
- 非本期范围（延后事项）
- 成功标准（业务、技术、运营）

### 3.3 阶段计划总览
按六大阶段拆分：
1. 基础数据管理
2. 种植与加工管理
3. 物流与销售管理
4. 系统与存证管理
5. 增值服务模块
6. 首页工作台

每阶段建议字段：
- 阶段目标
- 开始/结束时间
- 里程碑
- 依赖关系
- 阶段状态

### 3.4 功能模块拆解（WBS）
按“阶段 -> 模块 -> 功能点”拆解。

建议统一跟踪字段：
- 功能编号（如 P2-M1-F03）
- 所属阶段
- 模块名称
- 功能名称
- 优先级（P0/P1/P2）
- 负责人
- 开发状态（未开始/进行中/联调中/测试中/已完成/已上线）
- 计划开始/结束时间
- 实际开始/结束时间
- 进度百分比
- 验收标准
- 备注

### 3.5 验收与交付清单
建议按阶段给出：
- 功能验收标准
- 测试验收标准
- 数据验收标准
- 文档交付清单
- 上线条件

### 3.6 风险与问题台账
建议字段：
- 风险编号
- 风险描述
- 影响范围
- 风险等级（高/中/低）
- 应对策略
- 责任人
- 当前状态
- 关闭日期

### 3.7 变更记录
建议字段：
- 变更编号
- 变更内容
- 变更原因
- 影响评估（工期/成本/范围）
- 审批结果
- 生效时间

### 3.8 周报与里程碑回顾
建议每周更新：
- 本周完成
- 下周计划
- 阻塞项
- 需决策事项

## 4. 状态定义（统一口径）
- 未开始：未进入开发。
- 进行中：已进入开发阶段。
- 联调中：前后端/设备/第三方接口联调中。
- 测试中：功能测试、集成测试执行中。
- 已完成：通过验收，待上线或已上线。
- 已延期：超出计划时间且未完成。
- 已取消：需求确认取消。

## 5. 里程碑建议

### M1：基础模型与配置完成
- 基础数据管理相关配置能力可用。

### M2：种植与加工主流程跑通
- 环境接入、农事记录、批次关联、质检报告可闭环。

### M3：物流与销售闭环
- 物流追踪、在途预警、品牌溯源页面、扫码查询可用。

### M4：系统存证能力落地
- 用户权限、设备接入、数据存证与审计可用。

### M5：增值能力试点
- 碳足迹、风控指标、供应链金融接口可试运行。

### M6：首页工作台上线
- 看板、预警、任务待办、趋势统计可服务运营。

## 7. 更新机制
- 新任务启动前：先阅读“正式项目开发计划文档”。
- 每完成一个功能点：同步更新状态、完成时间、完成度、验收结果。
- 每周固定时间：更新周报、风险、变更。
- 每个阶段完成时：更新阶段完成度并提交阶段验收结论。

## 8. 与若依二开的衔接建议
- 若依已有用户、角色、权限、日志、代码生成能力，可直接复用。
- 业务模块按阶段落地，优先 P0 和 P1 功能。
- 设备接入、存证接口、第三方 API 建议采用“接口适配层”统一管理。
- AI 模型类能力先保留“接口占位 + 结果回写”机制，避免首期过度复杂。

## 9. 下一步建议
在本设计方案确认后，生成正式文件《启元农链_项目开发计划文档.md》，并将本方案中的模板转为可执行计划（补齐时间、负责人、验收口径）。

## 10. 根目录文档治理原则
- 根目录只保留项目启动与执行必需的 5 份 Markdown，其余专项检查报告、临时清单和一次性纪要都归入正式计划文档后删除。
- 正式计划文档是唯一的执行真相来源，设计方案只保留结构原则、字段口径和更新机制。
- 当新增功能说明容易和计划文档重复时，优先在计划文档中落状态，在设计方案中保留方法论摘要即可。
